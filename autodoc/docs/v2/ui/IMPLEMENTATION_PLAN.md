## 1. 목적

구현 순서의 원칙: **매 Sprint 끝에 눈으로 확인 가능한 동작물** + **v1 무수정 유지** + **여정(F1~F5) 단위 완결**. 기간은 1인 개발 가정의 추정치이며 파일럿 전 재산정한다.

## 3. UX 원칙

| 원칙 | 계획 반영 |
|---|---|
| 여정 단위 완결 | Sprint 경계 = 여정 완결점 (S3=F1, S5=F3·F4) — "반쯤 되는 기능 여럿" 금지 |
| 품질 하한은 마지막에 몰지 않기 | 접근성·오류 문법은 S1 부품 계약부터 내장 — S6은 점검이지 시작이 아님 |
| 눈으로 확인 | 매 Sprint 완료 조건에 사용자 관점 시나리오 포함 (기술 완료 ≠ 완료) |

## 4. 사용자 흐름 — Sprint 진행 규칙

```
Sprint 시작 → 범위 확인 (MVP_SCOPE 표 대조)
  ↓ 구현 (v1 무수정 검증을 매 커밋 습관화)
Sprint 종료 → 완료 조건 시연 (체크리스트)
  ├─ 통과 → 다음 Sprint
  └─ 미달 → 이월 항목 명시 후 진행 여부 결정 (조용한 이월 금지)
중대 발견(설계 결함) → 해당 UI 문서 개정 먼저, 구현은 후행
```

## 6. 확장성

- MVP 이후(Workflow·Graph·Plugin…)는 본 계획의 후속 Sprint가 아니라 **[../ROADMAP.md](../ROADMAP.md) 단계의 별도 계획**으로 작성 — 이 문서는 MVP 전용.
- Sprint 재편(인력 증가 등)은 완료 조건 불변·순서 의존만 준수하면 자유 (S1→S2→S3 순서와 S3→S5 의존은 고정).

## 7. 장점

1. **세 번의 완결점** — S3(작성 가치)·S5(학습 가치)·S7(실증)마다 중간 판단 기회.
2. **리스크 전진 배치** — 최난도(Preview·편집·Import)를 중반에 배치해 늦은 발견을 방지.
3. **범위 방어 장치 내장** — 매 Sprint가 MVP_SCOPE 표를 참조 — 슬금슬금 확장 차단.

## 8. 단점

1. **추정의 불확실성** — 18주는 1인 가정 추정치 — 실제 팀 구성·v1 코드 재사용률에 따라 ±40% 변동 가능. (→ S1·S2 실측 후 전면 재산정 약속)
2. **직렬 의존** — S3 지연이 전체를 민다. (→ S4 일부(입력 6종)는 S3와 병행 가능하게 분리 설계)
3. **파일럿 1회의 한계** — 3~5인 파일럿으로 전사 일반화는 무리. (→ S7 후 확대 파일럿을 로드맵 단계로)
